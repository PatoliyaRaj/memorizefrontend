## Problem Statement

The frontend `src/` directory needs:
1. **Proper centralization** of exports via index files
2. **Landing page feature** with hero, sections, testimonials
3. **Theme toggle** integrated in the main navbar/header
4. **State management** centralized with Zustand
5. **No breaking changes** to existing code

Current gap: No landing page exists; exports are scattered; state management setup incomplete.

---

## What Makes This Cool

A truly reusable landing page that:
- вњ… Works in dark/light mode automatically
- вњ… Exports cleanly via index files for easy imports
- вњ… Integrates theme toggle into navbar
- вњ… Scales as more pages are added
- вњ… Uses Zustand for UI state (nav open/close, theme)
- вњ… Zero disruption to existing features

---

## Constraints

- Must not break existing dashboard, study, auth features
- Theme toggle must work in navbar/header
- All exports should flow through index files
- Zustand stores must coexist with React Context
- Landing page should match Design.md specifications exactly

---

## Premises

1. **Landing page is a new feature, not a route group** вЂ” It lives at `/` (root), separate from `/(dashboard)`, `/(study)`, `/(auth)` groups
2. **Centralized exports via index files reduce import friction** вЂ” Better than deep path imports
3. **Zustand is the source of truth for UI state** вЂ” Theme, navbar open/close, mobile menu state
4. **ThemeProvider stays in layout.tsx** вЂ” Wraps the entire app at the root level
5. **Landing page components are feature-scoped** вЂ” Everything in `src/features/landing/`

## Approaches Considered

### Approach A: Minimal Reorganization (Fastest) вњ… RECOMMENDED
**Summary:** Add landing feature, create index files, integrate theme toggle into existing header. Minimal refactoring.

**Effort:** S (Small)  
**Risk:** Low  
**Pros:**
- вњ… Zero changes to existing features
- вњ… Can ship in 2-3 hours
- вњ… Landing feature is self-contained
- вњ… Easy to iterate on after launch

**Cons:**
- вќЊ Requires adding index files incrementally (not all at once)
- вќЊ Some existing imports stay as-is until refactored

**Reuses:**
- Existing ThemeProvider
- Existing Zustand store pattern
- Existing component library

---

### Approach B: Full Reorganization (Most Complete)
**Summary:** Reorganize entire `src/` with strict index file pattern, full Zustand store audit, landing feature. Everything refactored at once.

**Effort:** M (Medium)  
**Risk:** Medium (high chance of breaking something)  
**Pros:**
- вњ… Perfect long-term structure
- вњ… All exports centralized
- вњ… Consistent import patterns

**Cons:**
- вќЊ Requires refactoring existing code (risky)
- вќЊ Longer to ship
- вќЊ Possible import breakage

---

### Approach C: Hybrid (Best Balance)
**Summary:** Add landing feature + create index files for NEW code only. Leave existing features untouched. Refactor existing features individually as bugs/features are touched.

**Effort:** M (Medium)  
**Risk:** Low  
**Pros:**
- вњ… Landing page ready to ship immediately
- вњ… Clean index pattern for all new code
- вњ… Existing features stay stable
- вњ… Natural refactor path as codebase evolves

**Cons:**
- вќЊ Mixed patterns during transition (some old, some new)

---

**в†’ APPROACH C (Hybrid): Add Landing Feature + Index Files for New Code**

Why: Gives you a clean, shippable landing page today without risking existing features. New code follows best practices. Existing code refactors naturally as you touch it.

### 3. Zustand Nav Store

```typescript
// src/stores/use-nav-store.ts
import { create } from 'zustand'

interface NavState {
  isMobileMenuOpen: boolean
  toggleMobileMenu: () => void
  closeMobileMenu: () => void
}

export const useNavStore = create<NavState>((set) => ({
  isMobileMenuOpen: false,
  toggleMobileMenu: () => set((state) => ({ isMobileMenuOpen: !state.isMobileMenuOpen })),
  closeMobileMenu: () => set({ isMobileMenuOpen: false }),
}))
```
